## 1. Product summary

A live, AI-powered reflection activity for workshops and experiential learning. Participants join a session via a short code (no account), submit guided reflections pushed by the facilitator in real time, and at the end receive a personalised first-person learning summary generated by an LLM from their own words.

**One-line positioning:** A live AI-powered reflection activity that turns workshop moments into personalised, first-person learning insights.

**Core differentiator vs. an LMS:** facilitator-paced live experience + AI synthesis in the participant's own voice.

**Platform context.** Reflective Journal is one of several **learning activities** under the umbrella SaaS at `game.knywong.com`. It is not a standalone product. Facilitators sign in once to the platform (Google Sign-In) and access individual activities at sub-paths. Reflective Journal mounts at:

- `game.knywong.com/reflectivejournal` — participant entry point
- `game.knywong.com/reflectivejournal/facilitator` — facilitator Control Room

Each activity is its own folder under `C:\Users\knywo\learning-activities\` and its own Firebase project, following the pattern set by the existing `impact-bingo` activity.

## 2. Users

| User | Auth | Device | Primary need |
|---|---|---|---|
| Facilitator | Google Sign-In (Firebase Auth, Google provider) — same provider as the rest of the game.knywong.com platform | Laptop | Run a live workshop, push activities, see completion |
| Participant | None — anonymous Firebase auth, identity is just a nickname + workshop code | Phone | Reflect quickly, get a useful personal summary |

**Note on facilitator "single sign-in":** Each activity is its own Firebase project, so each has its own Firebase Auth user pool. The user experience still *feels* like single sign-in because Google OAuth re-consents instantly between projects — facilitator clicks "Sign in with Google" once per activity, ever, then the Google session carries through silently. This matches the pattern already used by `impact-bingo`.

Constraint: a participant must be able to join in **under 30 seconds** from URL tap to first reflection screen.

## 3. Product principles

1. **Low friction over feature depth** — every UI choice must justify itself against the join-in-30-seconds target.
2. **Facilitator paces the experience** — participants do not freely browse activities; they receive what the facilitator pushes.
3. **Participant voice is sacred** — AI summaries never invent content and never sound corporate.
4. **Mobile-first for participants, desktop-first for facilitators.**
5. **Reflection is the product** — analytics, sharing, exports are secondary.

## 4. Core flows

### 4.1 Participant flow

1. Open URL → enter **workshop code** + **nickname** → join.
2. Land on "Waiting for facilitator…" state.
3. Facilitator pushes an activity → participant's screen swaps to that activity within ~1s.
4. Participant submits response. Autosave fires on every change (debounced 500ms).
5. Steps 3–4 repeat for the session.
6. Facilitator pushes "Generate summary" → participant taps **Generate my learning summary**.
7. AI returns a 4-section first-person summary. Participant receives a **summary code** for later retrieval.

### 4.2 Facilitator flow

1. Land on `game.knywong.com/reflectivejournal/facilitator` (typically arrived from the game.knywong.com activity hub). If not already signed in to this Firebase project, click **Sign in with Google** → Google OAuth → land in Control Room. Subsequent visits skip the click.
2. Create workshop → system generates 6-character workshop code. Facilitator shares the participant URL (`game.knywong.com/reflectivejournal`) and the code with the room.
3. Enter Control Room. Sees live participant count, current activity, activity library.
4. Push activities one at a time. Monitor completion (count of submitted / total joined).
5. Push "Generate summary" at end.
6. Post-workshop: access archived session, export responses (post-MVP).

## 5. Activities

### 5.1 Activity model

Every activity is a document with:
- `type` — one of the supported types below
- `prompt` — the question or instructions shown to the participant
- `config` — type-specific config (e.g. columns for `table`)
- `privacyMode` — `private` | `visible` | `anonymous_wall` | `shareable`

### 5.2 MVP activity types

| Type | Response shape | Notes |
|---|---|---|
| `reflection` | `{ text: string }` | Single open-ended text area |
| `kiss` | `{ keep, improve, stop, start: string }` | 4 labelled text areas |
| `table` | `{ rows: Array<Record<columnKey, string>> }` | Columns defined in activity `config.columns` |
| `commitment` | `{ commitment: string, by: string? }` | Action-oriented |

### 5.3 Post-MVP activity types

`drawing`, `rating`, `vote`, `rank`, `image_upload`, `ai_conversation`, `audio`, `peer_reflection`. Schemas to be defined when scheduled in the roadmap.

### 5.4 Privacy modes

| Mode | Facilitator sees | Other participants see |
|---|---|---|
| `private` | Completion status only | Nothing |
| `visible` | Full responses | Nothing |
| `anonymous_wall` | Full responses | All responses, no names |
| `shareable` | Full responses | Only what each participant opts to share |

## 6. AI summary

### 6.1 Output structure

Exactly four sections, in this order, each 1–3 sentences:

1. **What I noticed about myself**
2. **What I learned from the activities**
3. **What I want to apply next**
4. **One sentence I can share**

### 6.2 Voice rules (hard constraints)

- First person ("I noticed…", "I want to…").
- Built **only** from the participant's own submitted text. No fabrication, no generic learning platitudes.
- Mirror the participant's register and vocabulary — if they wrote casually, the summary is casual.
- Forbidden phrasings: "leverage", "synergy", "stakeholder", "key takeaway", "growth journey", "in conclusion", "moving forward".
- If the participant submitted very little, the summary stays short. Never pad.
- Output is plain text, no headers, sections separated by blank lines and prefixed with the section label.

### 6.3 Prompt contract (input)

The AI is given:
- All response text the participant submitted during the session, in chronological order, each tagged with the activity prompt.
- The four section labels.
- The voice rules above.

The AI is **not** given the participant's name or the facilitator's notes.

### 6.4 Provider

**Google Gemini 2.0 Flash** via the AI Studio API, free tier. Chosen for the genuine free tier (Anthropic and OpenAI only offer signup credits). The summary call is server-side via Firebase Functions; the API key never reaches the client. Provider is abstracted behind a single `generateSummary(responses)` function so it can be swapped to Anthropic Claude (Sonnet 4.6 or later) once the product earns paid usage.

**Privacy note:** Google's free tier may use prompts for model training. Participant reflections are workshop content, not high-sensitivity data, but this should be disclosed in the participant join screen ("Your reflections are processed by Google AI to generate your summary"). If a facilitator runs a session with sensitive content, they should switch to paid Gemini or Anthropic — leave a TODO in the Functions config for a per-workshop provider override.

### 6.5 Regeneration policy

A participant can generate their summary **once**. After that the summary is read-only and stable — the summary code always resolves to the same text. **Exception:** if the participant submits new responses after generating, they may regenerate **one** more time to incorporate them. After that second generation the summary is locked. This keeps AI costs predictable and treats the reflection as a moment, not a slot machine.

## 7. Technical architecture

### 7.1 Stack

- **Frontend:** Next.js (App Router) with **static export** (`output: 'export'`), TypeScript, Tailwind CSS. Output is a folder of static files (HTML + JS + CSS), deployed via Firebase Hosting — same deployment model as `impact-bingo`. There is no server-rendered runtime.
- **Auth:** Firebase Authentication, **Google provider** for facilitators; **anonymous** for participants.
- **Database:** Cloud Firestore (chosen over Realtime Database for query flexibility; realtime via `onSnapshot` listeners).
- **Server functions:** Firebase Functions (Node) — used for the AI summary call and any operation that must not run client-side.
- **AI:** Google Gemini 2.0 Flash (free tier), called from Functions. Abstracted behind `generateSummary()` for swap-out.
- **Hosting:** Firebase Hosting. The activity's static build is deployed to the **`impact-bingo` Firebase project** (which owns the `game.knywong.com` custom domain) via the platform-level deploy described in §7.4. All activities share one Firebase project.

### 7.2 Realtime sync

- Participant screens subscribe via `onSnapshot` to:
  - their workshop document (to detect the current pushed activity)
  - their own response document for the active activity (for autosave confirmation)
- Facilitator Control Room subscribes to:
  - workshop document
  - all participant documents in the workshop (for join count + typing/completion status)
  - all response documents for the active activity (for live completion)

### 7.3 Autosave

- Client debounces input changes by 500ms, writes to the response document.
- The response document has a `submitted: boolean` flag flipped true only when the participant explicitly submits.

### 7.4 Deployment & platform integration

**Project layout.** The activity lives at `C:\Users\knywo\learning-activities\reflective-journal\` alongside other activities. Platform-level Firebase config (Hosting, Firestore rules, Functions, Emulators) lives one level up at `C:\Users\knywo\learning-activities\`. Each activity folder is app code only.

**Firebase project.** Single project `impact-bingo` for all activities. Firestore, Auth, Functions, and Hosting all belong to that project. Firestore collections are name-spaced per activity (workshops, responses, etc. are unique to this activity).

**Build → deploy (run `scripts/deploy.sh` from `learning-activities/`):**
1. Copies `impact-bingo/public/` → `learning-activities/public/impactbingo/`
2. `npm run build` in `reflective-journal/` → copies `out/` → `learning-activities/public/reflectivejournal/`
3. `firebase deploy --only hosting,firestore,functions` from `learning-activities/`

**Sub-path routing.** `game.knywong.com` is the custom domain for the `impact-bingo` project. The platform `firebase.json` at `learning-activities/` defines rewrites for all activity paths. See **D6 (resolved)** in §12.

**Per-activity firebase.json.** Each activity folder keeps a minimal `firebase.json` with emulator config only — used for local development and CI rules tests. It is not used for deployment.

**No SSR.** Because of static export, there is no server-side rendering. All data fetching happens in the browser via the Firebase client SDK. This is fine for our flows.

## 8. Firestore data model

```
facilitators/{uid}
  email: string                 // from Google profile (Firebase Auth)
  displayName: string           // from Google profile
  photoURL: string?             // from Google profile
  createdAt: timestamp          // first time this Google user signed in to this activity

workshops/{workshopId}
  code: string                  // 6-char uppercase alphanumeric, indexed for join lookup
  name: string
  facilitatorId: string         // -> facilitators/{uid}
  status: "draft" | "live" | "ended"
  currentActivityId: string?    // -> activities subcollection
  summaryUnlocked: boolean      // facilitator toggles when participants can generate
  createdAt: timestamp
  endedAt: timestamp?

workshops/{workshopId}/activities/{activityId}
  type: "reflection" | "kiss" | "table" | "commitment"
  prompt: string
  config: object                // type-specific
  privacyMode: "private" | "visible" | "anonymous_wall" | "shareable"
  order: number
  createdAt: timestamp

workshops/{workshopId}/participants/{participantId}
  nickname: string
  joinedAt: timestamp
  lastSeenAt: timestamp
  summaryCode: string?          // generated when summary is created

workshops/{workshopId}/responses/{responseId}
  participantId: string
  activityId: string
  data: object                  // matches activity type's response shape
  submitted: boolean
  updatedAt: timestamp
  submittedAt: timestamp?
  // Composite index on (participantId, activityId) — one response per (participant, activity)

workshops/{workshopId}/summaries/{summaryId}
  participantId: string
  code: string                  // unique, used for retrieval
  text: string                  // the 4-section AI output
  generatedAt: timestamp
  modelVersion: string          // e.g. "claude-sonnet-4-6"
```

### 8.1 Security rules (summary)

- `facilitators/{uid}` — readable/writable only by that uid. The uid comes from Firebase Auth's **Google provider** sign-in.
- `workshops/{id}` — the facilitator-owner (matched on `facilitatorId == request.auth.uid`, where the user signed in with Google) has full access. Participants can read only the fields needed to render their own screen (code, status, currentActivityId). Write access for participants is limited to creating their own participant doc and their own response docs.
- `responses/{id}` — a participant can read/write only responses where `participantId == request.auth.uid`. Participants are signed in via **Firebase anonymous auth** at join time, which gives every participant a stable uid with zero UX cost.
- `summaries/{id}` — readable by the owning participant (via anon uid) or by anyone with the matching `code`.

Note: rules must distinguish between **Google-signed-in users** (facilitators) and **anonymous-signed-in users** (participants). Use Firebase Auth's `firebase.auth.token.firebase.sign_in_provider` claim (`"google.com"` vs `"anonymous"`) to enforce this.

### 8.2 Workshop code format

6 characters, uppercase, alphanumeric, excluding ambiguous characters (`0`, `O`, `1`, `I`, `L`). Generated server-side, retried on collision.

## 9. Non-functional requirements

| Area | Requirement |
|---|---|
| Performance | Activity push → participant render < 1.5s p95 |
| Performance | Autosave write → confirmation < 800ms p95 |
| Mobile | Works on iOS Safari 15+ and Chrome Android last 2 versions |
| Offline | Reads from Firestore offline cache when connection drops; writes queue and flush on reconnect |
| Accessibility | All inputs labelled; contrast meets WCAG AA |
| Privacy | Participant nicknames are not personal data by policy; no email collection for participants in MVP |
| Cost | Single facilitator running a 30-person 2-hour workshop should cost < $1 in AI + infra |

## 13. Glossary

- **Workshop** — a single live session run by one facilitator.
- **Activity** — one reflection prompt pushed during a workshop.
- **Response** — one participant's answer to one activity.
- **Summary** — the AI-generated 4-section reflection produced at session end.
- **Summary code** — short code a participant uses to retrieve their summary later, without an account.
- **Control Room** — the facilitator's live dashboard during a workshop.
